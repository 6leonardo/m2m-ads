# Database Schema

## Files

Schema files live in `app/db/`. The convention is:

| File | Purpose |
|------|---------|
| `schema.1.sql` | Baseline — defines the full database from scratch |
| `schema.2.sql`, `schema.3.sql`, … | Migrations applied on top of the baseline |

`create-db.sh` applies all files in order.

---

## Versioning policy

### While in development (no production traffic)

Replace `schema.1.sql` in place. Wipe the database and recreate from scratch:

```bash
cd app/db
docker compose down
sudo rm -rf postgres_data init
docker compose up -d
./create-db.sh
```

There is no migration history to preserve. The `db_version` table records the
version string embedded in the applied file (e.g. `1.0.0`).

### After going live

**Never edit an existing schema file.** Add a new numbered file instead:

```
schema.2.sql   ← new migration
schema.3.sql   ← next migration
…
```

Each migration file must be idempotent using a `DO` block guard:

```sql
-- schema.2.sql — add widgets table
DO $$ BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM db_version WHERE version = '1.1.0'
  ) THEN

    CREATE TABLE widgets ( … );

    INSERT INTO db_version (version) VALUES ('1.1.0');
  END IF;
END $$;
```

`create-db.sh` applies all `schema.*.sql` files in lexicographic order, so
running it again on an already-migrated database is safe.

## Design notes

### UUID v7

All primary keys use UUID v7, generated by PostgreSQL 17's built-in
`gen_random_uuid()`. UUID v7 is time-sortable, so rows naturally sort by
creation time without a separate `created_at` index.

### Canonical match ordering

The `matches` table enforces `ad_id_1 < ad_id_2` (CHECK constraint). Because
UUID v7 sorts chronologically, `ad_id_1` is always the older ad. This prevents
duplicate rows for the same pair.

### Vector similarity

`announcements.embedding` stores a 1536-dimension vector (OpenAI
`text-embedding-3-small`). An IVFFlat index with `lists = 100` is created at
schema time; recall improves as data grows.
